# Документация Проекта: AI Life Simulator

## 1. Введение и Обзор

**AI Life Simulator** - это проект, моделирующий жизнь и поведение примитивных искусственных агентов (существ) на дискретной 2D-карте. Симуляция включает в себя:

*   **Два типа агентов:** "Мирные" (Peaceful) и "Хищники" (Predator).
*   **Взаимодействие со средой:** Перемещение по карте, взаимодействие с различными типами местности (земля, вода, препятствия).
*   **Основные потребности:** Голод, который со временем уменьшается и требует пополнения.
*   **Поведение:** Поиск пищи (мирные ищут еду, хищники охотятся на мирных), избегание опасностей (мирные избегают хищников), размножение при достатке ресурсов.
*   **Обучение:** Поведение агентов управляется нейронными сетями, которые обучаются с использованием методов обучения с подкреплением (Reinforcement Learning - RL) в фоновом режиме во время симуляции.
*   **Интерфейс:** Графический интерфейс пользователя (GUI) на базе Pygame, позволяющий наблюдать за симуляцией, редактировать карту и управлять процессом обучения.

Цель проекта - создать визуально наблюдаемую среду, где можно экспериментировать с обучением агентов сложному поведению выживания и взаимодействия.

## 3. Ключевые Технологии

*   **Python 3.12+:** Основной язык программирования.
*   **Pygame:** Библиотека для создания 2D-игр и графического интерфейса.
*   **PyTorch:** Фреймворк для глубокого обучения, используется для нейронных сетей агентов.
*   **NumPy:** Библиотека для эффективных численных операций, особенно для работы с картой и состояниями.

## 4. Архитектура

Проект разделен на два основных слоя:

1.  **Frontend (Пользовательский Интерфейс):**
    *   Реализован с помощью **Pygame**.
    *   Отвечает за отображение игрового мира (карты, агентов), элементов управления (кнопки, переключатели, палитра).
    *   Обрабатывает ввод пользователя (клики мыши, нажатия клавиш).
    *   Запускается и управляется из корневого `main.py` через класс `GameEngine`.
    *   Взаимодействует с Backend через `SimulationAPI`.
    *   Хранит *видимое пользователю* состояние игры в `GameState`.
    *   Отображает карту с помощью `MapView`, который позволяет масштабировать, перемещать и редактировать карту.

2.  **Backend (Логика Симуляции и AI):**
    *   Сосредоточен в модуле `ai_life_simulator._backend.agents`.
    *   Отвечает за всю логику симуляции: состояние среды, движение агентов, взаимодействия, правила мира (голод, размножение, смерть).
    *   Реализует агентов (`PeacefulAgent`, `PredatorAgent`) и среду (`Environment`).
    *   Содержит нейросетевые модели (`EmbeddingDuelingAgentNN`) для управления поведением агентов.
    *   Реализует алгоритмы обучения с подкреплением (DDQN + PER) для тренировки моделей.
    *   Работает в **отдельном потоке** (`SimulationWorker`), чтобы не блокировать GUI во время вычислений и обучения.
    *   Предоставляет интерфейс для взаимодействия с Frontend (`SimulationAPI`).

3.  **Взаимодействие Frontend и Backend:**
    *   **`SimulationAPI`:** Класс-посредник. Frontend создает его экземпляр и вызывает методы (`start`, `stop`, `pause`, `resume`, `get_map_snapshot`, etc.).
    *   **`SimulationWorker`:** Backend-поток, управляемый `SimulationAPI`. Получает команды от API через потокобезопасную очередь (`command_queue`). Отправляет ответы (`response_queue`).
    *   **Общая Карта (`shared_map_for_viz`):** Backend (`SimulationWorker`) периодически обновляет этот NumPy-массив, отражая текущее состояние мира. Frontend (`GameEngine` -> `MapView`) читает этот массив для отрисовки. Это одностороннее обновление (Backend -> Frontend) для визуализации.
    *   **Редактирование Карты:** Frontend (`MapView`) изменяет свое локальное представление карты (`map_data` в `GameState`). При старте симуляции/обучения эта карта передается в `SimulationAPI` и используется для инициализации `Environment` в Backend. Во время работы симуляции редактирование карты в GUI блокируется (`ban_brush_mode`).

## 5. Описание Модулей и Классов

### 5.1. Корневой `main.py`

*   **Назначение:** Точка входа для запуска GUI приложения.
*   **Логика:**
    *   Импортирует `GameEngine` и необходимые библиотеки (`pygame`, `sys`).
    *   Определяет базовую конфигурацию (`config`) для GUI (размеры, FPS, отступы).
    *   Инициализирует `GameEngine`.
    *   Запускает главный цикл Pygame (`while mainLoop`):
        *   Обрабатывает события (выход, Escape, события от UI-элементов, события от карты). *Важно:* События сначала передаются UI-элементам (в обратном порядке отрисовки), затем карте. Если элемент обработал событие, оно дальше не идет.
        *   Очищает экран (`game.window.fill`).
        *   Рисует элементы UI на отдельной поверхности (`game.ui_surface`).
        *   Копирует UI-поверхность на основное окно (`game.window.blit`).
        *   Рисует карту (`game.map_view.draw`) поверх UI.
        *   Контролирует FPS (`clock.tick`).
    *   Завершает работу Pygame при выходе из цикла.

### 5.2. `ai_life_simulator._front.assets` (`assets.py`)

*   **Назначение:** Хранение констант и утилита для загрузки ресурсов.
*   **Константы:**
    *   Пути к файлам (шрифты, иконки, логи, модели).
    *   Цвета (WHITE, BLACK, FIELD_COLOR, UI-цвета и т.д.).
    *   Параметры (уровень логирования, доступность CUDA).
*   **`StaticLoader`:**
    *   **`__init__`:** Инициализирует логгер, загружает и устанавливает иконку окна.
    *   **`get...Colors`:** Возвращает словари с цветовыми схемами для разных состояний UI-элементов (кнопки, переключатели, палитра).
    *   **`getPaletteCodes`:** **Ключевой метод.** Возвращает словарь для двустороннего преобразования: Цвет -> [Код, Имя] и Код -> Цвет. Используется для связи между цветом в палитре/на карте и числовым кодом объекта в Backend.
    *   **`get...Icon`:** Загружает иконки для режимов карты (перетаскивание, рисование).
    *   **`getLogger`:** Статический метод для получения экземпляра логгера.
    *   **`getPixelFont`:** Загружает пиксельный шрифт указанного размера.

### 5.3. `ai_life_simulator._front.game_logic` (`game_logic.py`)

*   **Назначение:** Управление общей логикой игры/приложения со стороны Frontend.
*   **`GameState`:**
    *   Хранит динамическое состояние, видимое или управляемое из GUI.+
    *   `map_data`: NumPy массив, представляющий карту, которую видит и редактирует пользователь. **Важно:** Этот массив передается в Backend при старте и обновляется из Backend (через `map_loader` в `GameEngine`), если симуляция запущена.
    *   `current_color`: Текущий выбранный цвет/инструмент для рисования на карте.
    *   `is_running`: Флаг, указывающий, запущена ли симуляция в Backend.
    *   `map_maybe_updated`: Флаг для потока `map_loader`, указывающий, что нужно продолжать обновлять `map_data` из Backend.
    *   `compute_mode`: Режим вычислений ("CPU" или "GPU"), выбранный пользователем.
    *   `set_map_data()`: Обновляет ячейку карты `map_data` на основе выбранного цвета/инструмента.
*   **`GameEngine`:**
    *   **`__init__`:**
        *   Принимает `config` из `main.py`.
        *   Настраивает DPI Awareness (для Windows).
        *   Инициализирует Pygame.
        *   Создает `StaticLoader` и `GameState`.
        *   Настраивает конфигурацию для Backend (`agents_state: Config`).
        *   Создает окно Pygame (`self.window`).
        *   Создает поверхность для UI (`self.ui_surface`).
        *   Загружает шрифты.
        *   **Создает UI:** Вызывает `_create_ui_background_elements` (статичные фоны зон) и `_create_ui_elements` (интерактивные виджеты: кнопки, палитра, переключатель, панель обучения, `MapView`). Связывает действия кнопок с методами `GameEngine`.
    *   **`_setup_dpi`, `_setup_display`, `_create_ui_surface`, `_create_ui_background_elements`, `_load_fonts`:** Вспомогательные методы инициализации.
    *   **`_create_ui_elements`:** Ключевой метод создания всех виджетов интерфейса, их размещения и привязки действий (`action`). Использует разметку из `config` для расчета размеров и позиций.
    *   **Методы действий (`..._action`)**:
        *   `close_game_action`: Завершает приложение, останавливая Backend, если он запущен.
        *   `stop_action`: Вызывается кнопкой "Stop", останавливает обучение в Backend (`kill_func`), меняет статус панели.
        *   `reset_game_action`: Сбрасывает `map_data` в `GameState` (заполняет нулями), если симуляция не запущена.
        *   `select_tool_action`: Вызывается палитрой, обновляет `current_color` в `GameState` и цвет кисти в `MapView`.
        *   `set_compute_mode_action`: Вызывается переключателем CPU/GPU, обновляет `compute_mode` в `GameState` и `DEVICE` в `agents_state`.
        *   `toggle_training_action`: Вызывается кнопкой "Start Training". Если `start_training`=True, запускает `start_func` в отдельном потоке, обновляет `is_running` в `GameState`.
    *   **`kill_func`, `start_func`:** Методы для взаимодействия с `SimulationAPI` в отдельных потоках (старт/стоп обучения). `start_func` также запускает `map_loader`.
    *   **`map_loader`:** Метод, работающий в отдельном потоке, когда симуляция запущена. Постоянно (с задержкой `time.sleep(0.1)`) запрашивает снимок карты у `SimulationAPI` (`sim_api.get_map_snapshot()`) и обновляет `GameState.map_data`.
    *   **`get_palette_items_data`:** Формирует данные для палитры.

### 5.4. `ai_life_simulator._front.front` (`front.py`)

*   **Назначение:** Реализация конкретных UI-виджетов и вида карты.
*   **`UIElement`:** Базовый класс для всех элементов UI. Определяет `rect`, `visible`, `hovered`, базовые `handle_event`, `on_click`, `draw`.
*   **`Button`:** Наследуется от `UIElement`. Кнопка с текстом, цветами для разных состояний (normal, hover, active), обработкой нажатия (с визуальным сдвигом `click_offset`), привязкой действия (`action`).
*   **`PaletteItem`:** Наследуется от `UIElement`. Один элемент (квадрат) в палитре. Хранит тип (`item_type`), цвет, иконку (если есть), флаг `selected`. Рисует рамку при выборе (`selected_border_color`).
*   **`Palette`:** Наследуется от `UIElement`. Контейнер для `PaletteItem`. Размещает элементы сеткой (`cols`, `item_size`, `margin`). Обрабатывает клики на элементах, вызывает `on_select` при выборе нового элемента, подсвечивает выбранный элемент.
*   **`Switch`:** Наследуется от `UIElement`. Переключатель. Имеет два состояния (`state`: True/False), метки для состояний (`labels`), цвета фона и "ручки" (`colors`), действие при переключении (`action`). Отображает текущую метку под переключателем.
*   **`Label`:** Наследуется от `UIElement`. Простой текстовый элемент. Имеет текст, шрифт, цвет текста, опциональный цвет фона, выравнивание.
*   **`TrainingPanel`:** Наследуется от `UIElement`. Композитный виджет, содержащий кнопку (`Button` для старта) и метку (`Label` для статуса). Управляет состоянием "обучение активно/неактивно" (`training_active`).
*   **`MapView`:** **Ключевой класс Frontend.**
    *   **Назначение:** Отображение и взаимодействие с игровой картой.
    *   **Инициализация (`__init__`):**
        *   Принимает `viewport_rect` (область на экране для карты), размеры карты в тайлах, `loader`, и `map_storage` (ссылку на `GameState.map_data`).
        *   Настраивает параметры зума (`min_tile_size`, `max_tile_size`, `tile_size`). Рассчитывает `effective_min_tile_size`, чтобы карта всегда заполняла viewport.
        *   Инициализирует переменные камеры (`camera_x`, `camera_y`), состояния перетаскивания/рисования (`dragging`, `potential_click`, `mode`).
        *   Создает UI элементы внутри карты: кнопки выбора режима (перетаскивание/рисование) и слайдер зума.
    *   **Режимы:** `MODE_DRAG` (перемещение карты), `MODE_PAINT` (рисование на карте выбранным цветом).
    *   **Зум:** Управляется колесом мыши и слайдером. Метод `_update_zoom_dependent_vars` пересчитывает зависимые от `tile_size` параметры (размеры карты в пикселях, максимальное смещение камеры) и корректирует позицию камеры. Методы `_y_to_value` и `_value_to_y` преобразуют позицию слайдера в `tile_size` и обратно.
    *   **Перемещение (`MODE_DRAG`):** При зажатии ЛКМ и движении мыши изменяет `camera_x`, `camera_y`.
    *   **Рисование (`MODE_PAINT`):** При клике или зажатии ЛКМ и движении мыши вызывает `get_tile_indices_at_pos` для определения координат тайла под курсором и вызывает `GameState.set_map_data` (косвенно, через обновление `self.map_data`, который является ссылкой на `GameState.map_data`) для изменения его состояния на `draw_color`. Имеет флаг `paint_locked`, который устанавливается из `GameEngine`, когда симуляция запущена.
    *   **`handle_event`:** Обрабатывает события мыши (нажатия, отпускания, движение, колесо) для управления зумом (слайдер, колесо), переключением режимов (кнопки), перетаскиванием и рисованием. Различает клик и перетаскивание с помощью `CLICK_MOVE_THRESHOLD`.
    *   **`draw`:** Отрисовывает видимую часть карты (тайлы) на переданной поверхности (`surface`), учитывая `camera_x`, `camera_y` и `tile_size`. Использует `surface.set_clip` для отрисовки только в пределах `viewport_rect`. Также вызывает `_draw_ui` для отрисовки кнопок режимов и слайдера поверх карты.
    *   **`set_current_color`:** Устанавливает цвет для рисования (`draw_color`).
    *   **`ban_brush_mode`/`unlock_brush_mode`:** Блокирует/разблокирует режим рисования.

### 5.5. `ai_life_simulator._backend.agents` (`agents.py`)

*   **Назначение:** Ядро симуляции и искусственного интеллекта. Содержит всю логику поведения агентов, их обучения и взаимодействия со средой. Работает в отдельном потоке.
*   **`Config`:** Dataclass, хранящий все параметры симуляции и обучения (размеры карты, параметры агентов, награды, параметры RL, пути сохранения и т.д.). Экземпляр этого класса передается в `SimulationWorker`.
*   **Коды Объектов:** Константы (SURFACE_CODE, PEACEFUL_CODE и т.д.) для представления типов объектов на карте. Используются как индексы в `nn.Embedding`.
*   **RL Компоненты (PER):**
    *   **`Experience`:** Namedtuple для хранения одного шага опыта (state, action, reward, next_state, done).
    *   **`SumTree`:** Структура данных (двоичное дерево), используемая для эффективного сэмплирования с приоритетами в PER. Хранит приоритеты в листьях и суммы в узлах. `add`, `update`, `get_leaf`, `total`.
    *   **`PrioritizedReplayBuffer`:** Реализация буфера воспроизведения с приоритетами. Использует `SumTree`. `push` (добавляет опыт с приоритетом, рассчитанным по TD-ошибке), `sample` (выбирает батч с учетом приоритетов и возвращает опыт, индексы и веса важности (IS weights)), `update_priorities` (обновляет приоритеты после шага обучения). Параметры `alpha` (степень приоритизации) и `beta` (степень компенсации IS weights).
*   **Нейросетевая Модель (`EmbeddingDuelingAgentNN`):**
    *   **Архитектура:**
        1.  **Вход:** Локальное зрение агента (`local_view_indices`: матрица кодов объектов) и вектор состояния агента (`agent_state`: например, нормализованный голод).
        2.  **`nn.Embedding`:** Преобразует дискретные коды объектов из `local_view_indices` в эмбеддинги.
        3.  **CNN:** Несколько сверточных слоев (`Conv2d` + `ReLU`) обрабатывают эмбеддированное локальное зрение для извлечения пространственных признаков.
        4.  **Flatten + Dense:** Результат CNN выпрямляется и пропускается через полносвязный слой (`fc1_features`).
        5.  **State Processing:** Вектор состояния агента (`agent_state`) обрабатывается отдельным небольшим полносвязным слоем (`fc1_state`).
        6.  **Concatenate + Dense:** Признаки от CNN и обработанный вектор состояния объединяются и пропускаются через еще один полносвязный слой (`fc_combined_features`).
        7.  **Dueling Heads:** Выход разделяется на два "потока":
            *   **Value Head (`fc_value`):** Оценивает ценность состояния (V(s)). Выход - одно число.
            *   **Advantage Head (`fc_advantage`):** Оценивает преимущество каждого действия в данном состоянии (A(s, a)). Выход - вектор размером `num_actions`.
        8.  **Combine:** Q-значения для каждого действия собираются по формуле: `Q(s, a) = V(s) + (A(s, a) - mean(A(s, .)))`. Вычитание среднего преимущества стабилизирует обучение.
    *   **Назначение:** Оценивает Q-значения (ожидаемую долгосрочную награду) для каждого возможного действия агента в текущем состоянии. Используется для выбора действий и в процессе обучения.
*   **Классы Агентов:**
    *   **`Agent`:** Базовый dataclass для всех агентов. Хранит `id`, `pos` (позиция y, x), `hunger`, `type_code`, ссылку на `config`, флаг `alive`, `steps_in_water`. Метод `get_state_vector` возвращает вектор состояния (пока только голод).
    *   **`PeacefulAgent`, `PredatorAgent`:** Наследуются от `Agent`, просто устанавливают соответствующий `type_code`.
*   **`Environment`:**
    *   **Назначение:** Представляет мир симуляции, управляет его состоянием и взаимодействиями.
    *   **`__init__`:** Принимает начальную карту (`initial_map_with_entities`) и `config`. Инициализирует `base_map` (статичная карта: земля, вода, препятствия), `shared_map` (динамическая карта со всеми объектами), словари для агентов (`agents`, `peaceful_agents`, `predator_agents`), множество с позициями еды (`food_locations`). Вызывает `_initialize_environment_state` для парсинга начальной карты.
    *   **`_initialize_environment_state`:** Заполняет `base_map`, создает объекты агентов и еды на основе кодов в переданной карте, помещает их в `shared_map` и соответствующие структуры данных.
    *   **`reset`:** Сбрасывает состояние среды к начальному, используя `initial_map_ref`.
    *   **`_spawn_food`:** С некоторой вероятностью (`FOOD_SPAWN_RATE`) добавляет еду на случайную свободную клетку земли, если еды меньше максимума (`MAX_FOOD`).
    *   **`_dist_sq`, `_find_closest`:** Вспомогательные функции для расчета расстояний.
    *   **`get_padded_map`:** Возвращает `shared_map` с добавленными "рамками" из `PAD_VALUE` (код препятствия) для упрощения взятия локального зрения у края карты.
    *   **`get_local_view`:** Извлекает квадратную область (`VIEW_SIZE` x `VIEW_SIZE`) из `padded_map` вокруг позиции агента.
    *   **`get_agent_state`:** Возвращает кортеж (локальное зрение, вектор состояния) для заданного агента.
    *   **`step(actions)`:** **Центральный метод симуляции.** Принимает словарь `{agent_id: action}`. Выполняет один шаг симуляции:
        1.  **Обновление статуса и расчет намерений:** Уменьшает голод агентов, проверяет на смерть от голода. Рассчитывает *предполагаемую* новую позицию (`agent_intended_positions`) на основе действия. Сохраняет состояния *до* движения (позиции агентов, еды) для расчета наград за приближение/удаление.
        2.  **Разрешение конфликтов и определение финальных позиций:** Проверяет, не пытается ли агент выйти за карту или войти в статическое препятствие (`base_map`). Разрешает конфликты, если *несколько* агентов хотят перейти на *одну и ту же* свободную клетку (случайный агент остается, остальные стоят). Определяет итоговые позиции агентов (`final_positions`).
        3.  **Обработка взаимодействий:** На основе `final_positions` проверяет:
            *   **Утопление:** Увеличивает `steps_in_water`, если агент в воде; убивает, если превышен лимит.
            *   **Поедание еды (Мирный):** Если мирный агент оказался на клетке с едой, он ест (увеличивает голод, получает награду), еда удаляется (`food_eaten_coords`).
            *   **Поедание добычи (Хищник):** Если хищник оказался на клетке, где *в начале шага* был мирный агент (и этот мирный еще жив и не съеден другим хищником в этом шаге), хищник ест (увеличивает голод, получает награду), мирный агент помечается к удалению (`agents_to_remove`, `prey_eaten_ids`) и получает штраф за смерть.
        4.  **Обновление карты:** Применяет изменения к `shared_map`: удаляет съеденную еду и агентов, перемещает живых агентов на их `final_positions`, очищая их стартовые клетки (если они не были заняты кем-то другим).
        5.  **Расчет финальных наград и размножение:**
            *   Рассчитывает награды за приближение к еде/добыче или удаление от хищников, сравнивая расстояния *до* и *после* шага.
            *   **Размножение (Мирный):** Если у мирного агента достаточно голода (`REPRODUCTION_THRESHOLD`) и есть место рядом на свободной клетке земли, он создает нового агента (родитель теряет часть голода, получает награду), новый агент добавляется в симуляцию.
        6.  **Очистка:** Удаляет агентов, помеченных к удалению (`agents_to_remove`), из всех структур данных и с `shared_map`. Спавнит новую еду (`_spawn_food`).
        7.  **Формирование `next_states`:** Для всех агентов, которые *действовали* в этом шаге и остались живы, получает их новое состояние (`local_view` и `state_vector` с обновленной карты) и записывает в `next_states`. Для умерших `next_state = None`.
        8.  **Возврат:** Возвращает словари `next_states`, `rewards`, `dones` (только для тех `agent_id`, которые были в исходном `actions`).
*   **`SimulationWorker`:**
    *   **Назначение:** Управляет полным циклом симуляции и обучения в отдельном потоке.
    *   **`__init__`:** Создает `Environment`, инициализирует модели NN (основные и целевые), оптимизаторы (Adam), буферы PER (`peaceful_buffer`, `predator_buffer`), события для синхронизации потоков (`_stop_event`, `_pause_event`), очереди команд/ответов.
    *   **`run`:** Основной цикл рабочего потока:
        *   Обрабатывает команды из `command_queue` (pause, resume, save, stop, get_epoch, get_step).
        *   Проверяет паузу (`_pause_event`).
        *   **Проверка на вымирание:** Если один из типов агентов вымер, сбрасывает среду (`environment.reset`), очищает буферы.
        *   **Шаг симуляции:**
            *   Получает состояния живых агентов (`environment.get_agent_state`).
            *   Выбирает действия для них (`_select_action` с использованием epsilon-greedy).
            *   Выполняет шаг в среде (`environment.step`).
            *   Сохраняет полученный опыт (`state`, `action`, `reward`, `next_state`, `done`) в соответствующий PER буфер (`peaceful_buffer` или `predator_buffer`). *Важно:* Ошибка для расчета приоритета изначально `None`, она будет вычислена при первом обучении на этом опыте.
        *   **Обучение:** Если выполнены условия (прошло достаточно шагов `global_step`, текущий шаг кратен `TRAIN_FREQ`), вызывает `_train_step` для каждого типа агентов.
        *   **Обновление целевых сетей:** Периодически (каждые `TARGET_UPDATE_FREQ` шагов) копирует веса из основных моделей в целевые (`_update_target_networks`).
        *   **Обновление shared_map:** Периодически копирует состояние `environment.shared_map` в `shared_map_ref` для визуализации в GUI (с ограничением частоты).
        *   **Логирование:** Выводит статистику по эпохам (средние награды, потери, количество агентов, размер буфера).
        *   Инкрементирует `global_step`.
    *   **`_select_action`:** Реализует epsilon-greedy стратегию выбора действия: с вероятностью `epsilon` выбирает случайное действие, иначе - действие с максимальным Q-значением от основной модели. `epsilon` линейно уменьшается со временем (`_get_epsilon`).
    *   **`_train_step`:** Выполняет один шаг градиентного спуска для модели:
        *   Сэмплирует батч из PER буфера (`buffer.sample`), получает `experiences`, `indices`, `is_weights`.
        *   Преобразует батч в тензоры PyTorch (`_preprocess_batch`).
        *   Вычисляет текущие Q-значения `Q(s, a)` для действий из батча (используя основную модель).
        *   Вычисляет целевые Q-значения `target_Q` по формуле Double DQN: `r + gamma * Q_target(s', argmax_a' Q(s', a'))`. **Важно:** Действие `a'` выбирается основной моделью (`model`), а его ценность оценивается целевой моделью (`target_model`).
        *   Вычисляет TD-ошибку: `delta = target_Q - Q(s, a)`.
        *   Вычисляет взвешенную Huber Loss: `Loss = mean(IS_weights * HuberLoss(delta))`.
        *   Делает шаг оптимизатора (`optimizer.step`).
        *   Обновляет приоритеты в PER буфере на основе вычисленных TD-ошибок (`buffer.update_priorities`).
    *   **`_update_target_networks`:** Копирует веса `model.state_dict()` в `target_model`.
    *   **`save_models`, `load_models`:** Сохраняет/загружает веса моделей PyTorch (`.pth` файлы). `load_models` вызывается при старте `SimulationAPI`.
*   **`SimulationAPI`:**
    *   **Назначение:** Публичный интерфейс для управления Backend из основного потока (GUI).
    *   **`__init__`:** Принимает начальную карту от пользователя и `config`. Создает `shared_map_for_viz` (куда Worker будет писать обновления для GUI).
    *   **`start`:** Создает `SimulationWorker`, пытается загрузить модели (`worker.load_models`), запускает `worker.run` в отдельном потоке (`worker_thread`).
    *   **`stop`, `pause`, `resume`, `save_models`, `get_epoch`, `get_step`:** Отправляют соответствующие команды в `command_queue` рабочего потока (`_send_command`). `stop` дополнительно ждет завершения потока (`worker_thread.join`).
    *   **`get_map_snapshot`:** Возвращает *копию* текущего состояния `shared_map_for_viz`.
    *   **`is_running`:** Проверяет, активен ли рабочий поток.

## 6. Логика Взаимодействия Агентов (Детально в `Environment.step`)

1.  **Движение:**
    *   Агенты пытаются выполнить одно из 5 действий: стоять на месте (0), вверх (1), вниз (2), влево (3), вправо (4).
    *   **Ограничения:**
        *   Нельзя выйти за пределы карты.
        *   Нельзя войти в статическое препятствие (код `OBSTACLE_CODE` на `base_map`).
        *   Если два или более агента пытаются перейти на *одну и ту же* пустую клетку в одном шаге, только один (выбранный случайно) перемещается, остальные остаются на месте.
    *   **Результат:** Агенты занимают свои финальные позиции для этого шага.

2.  **Взаимодействия (проверяются на финальных позициях):**
    *   **С водой:** Если агент попадает на клетку с водой (`WATER_CODE` на `base_map`), счетчик `steps_in_water` увеличивается. При достижении лимита (`PEACEFUL_DEATH_IN_WATER_STEPS` или `PREDATOR_DEATH_IN_WATER_STEPS`) агент умирает (штраф `REWARD_DEATH`).
    *   **Поедание еды (Мирный):** Если мирный агент оказывается на клетке с едой (`FOOD_CODE` на `shared_map`), он съедает ее. Голод увеличивается (но не выше максимума), агент получает награду `REWARD_EAT_FOOD`, еда исчезает с карты.
    *   **Охота (Хищник):** Если хищник оказывается на клетке, где в *начале* этого шага находился мирный агент (`PEACEFUL_CODE` на `shared_map` до обновлений этого шага), хищник съедает мирного. Голод хищника увеличивается, он получает награду `REWARD_EAT_PREY`. Мирный агент умирает и получает штраф `REWARD_DEATH * 1.1`. Один мирный агент может быть съеден только одним хищником за шаг.

3.  **Размножение (Мирный):**
    *   Происходит *после* всех перемещений и взаимодействий.
    *   **Условия:** Мирный агент должен быть жив, его голод должен быть выше `REPRODUCTION_THRESHOLD`, общее число мирных агентов меньше `MAX_AGENTS_PER_TYPE`.
    *   **Процесс:** Агент ищет свободную соседнюю клетку земли. Если находит, он создает нового мирного агента на этой клетке. Голод родителя уменьшается вдвое, голод потомка устанавливается как доля от уменьшенного голода родителя. Родитель получает награду `REWARD_REPRODUCE`.

4.  **Смерть:**
    *   **Причины:** Голод (hunger <= `STARVATION_LIMIT`), утопление, поедание хищником.
    *   **Последствия:** Агент удаляется из симуляции, его клетка на карте очищается (если ее не занял другой агент), агент (или его опыт) получает штраф `REWARD_DEATH` (или чуть больше при поедании).

5.  **Награды и Штрафы:**
    *   **Шаг:** Небольшой штраф `REWARD_STEP` за каждый шаг (стимулирует эффективность).
    *   **Еда:** Награда `REWARD_EAT_FOOD` для мирного.
    *   **Охота:** Награда `REWARD_EAT_PREY` для хищника.
    *   **Размножение:** Награда `REWARD_REPRODUCE` для мирного.
    *   **Смерть:** Большой штраф `REWARD_DEATH`.
    *   **Приближение/Удаление:** Дополнительные награды рассчитываются *после* основного шага на основе изменения расстояния до ближайших объектов:
        *   Мирный: + награда за приближение к еде (`REWARD_APPROACH_FOOD_FACTOR`), + награда за удаление от хищника (`REWARD_FLEE_PREDATOR_FACTOR`).
        *   Хищник: + награда за приближение к мирному (`REWARD_APPROACH_PREY_FACTOR`).
        *   Награда пропорциональна `sqrt(dist_before) - sqrt(dist_after)`, чтобы стимулировать большее изменение на близких дистанциях.

## 7. Логика Обучения Модели (Детально в `SimulationWorker`)

Используется комбинация современных техник обучения с подкреплением для обучения агентов принятию решений (выбору действий) на основе их локального восприятия и внутреннего состояния (голода).

1.  **Алгоритм: Double Deep Q-Networks (DDQN)**
    *   **Основа:** DQN - использование нейронной сети для аппроксимации Q-функции `Q(s, a)`, которая предсказывает суммарную будущую награду при выполнении действия `a` в состоянии `s`.
    *   **Проблема DQN:** Стандартный DQN склонен к переоценке Q-значений из-за использования одной и той же сети для выбора лучшего действия и оценки его ценности на следующем шаге.
    *   **Решение DDQN:** Используются две сети:
        *   **Основная сеть (`model`):** Выбирает *лучшее* действие для следующего состояния `s'` (`argmax_a' Q_model(s', a')`).
        *   **Целевая сеть (`target_model`):** Оценивает *ценность* этого выбранного действия (`Q_target(s', argmax_a' Q_model(s', a'))`).
    *   **Целевое Q-значение:** `target = r + gamma * Q_target(s', argmax_a' Q_model(s', a'))`.
    *   **Целевая сеть:** Ее веса периодически (`TARGET_UPDATE_FREQ`) копируются из основной сети, что делает обучение более стабильным.

2.  **Буфер Опыта: Prioritized Experience Replay (PER)**
    *   **Основа:** Replay Buffer - хранение опыта `(s, a, r, s', done)` и сэмплирование из него мини-батчей для обучения. Это разрывает корреляции между последовательными шагами и повышает эффективность использования данных.
    *   **Проблема стандартного Replay Buffer:** Все переходы сэмплируются с равной вероятностью, даже если некоторые из них гораздо важнее для обучения (например, получение большой награды или штрафа).
    *   **Решение PER:** Переходам назначается приоритет, пропорциональный абсолютной величине ошибки временных различий (TD-ошибки): `priority = (|TD_error| + epsilon)^alpha`.
        *   `TD_error = target_Q - Q(s, a)`.
        *   `epsilon`: Малая константа для ненулевого приоритета.
        *   `alpha`: Параметр (0 до 1), контролирующий степень приоритизации (0 - равномерно, 1 - полностью по TD-ошибке).
    *   **Сэмплирование:** Используется `SumTree` для эффективного сэмплирования пропорционально приоритетам.
    *   **Коррекция смещения (Importance Sampling - IS):** Так как мы меняем распределение данных при сэмплировании, для коррекции смещения градиента используются веса важности: `IS_weight = (N * P(i))^(-beta)`.
        *   `N`: Размер буфера.
        *   `P(i)`: Вероятность сэмплирования перехода `i`.
        *   `beta`: Параметр (0 до 1), контролирующий степень коррекции (со временем увеличивается от `beta_start` до 1).
    *   **Применение:** Градиенты при обучении умножаются на `IS_weight` (в коде это делается умножением функции потерь на `is_weights_t`).

3.  **Архитектура Сети: Dueling DQN + Embedding**
    *   **Dueling DQN:** Как описано в п. 5.5, сеть разделяет оценку ценности состояния (`V(s)`) и преимущества действий (`A(s, a)`). Это помогает сети лучше понимать, какие состояния ценны сами по себе, независимо от действий.
    *   **Embedding Layer:** Так как входное локальное зрение (`local_view_indices`) состоит из дискретных кодов объектов (земля, вода, агент, еда...), слой `nn.Embedding` преобразует эти коды в плотные векторы признаков. Это позволяет сети эффективно работать с категориальными входными данными карты, передавая их затем в сверточные слои (CNN).

4.  **Процесс Обучения (`SimulationWorker._train_step`)**
    *   Происходит каждые `TRAIN_FREQ` шагов симуляции, если в буфере достаточно опыта (`MIN_BUFFER_SIZE_FOR_TRAIN`).
    *   Из PER буфера извлекается батч опыта вместе с индексами и IS-весами.
    *   Для батча вычисляются:
        *   Текущие Q-значения `Q(s, a)` (основная сеть).
        *   Целевые Q-значения `target_Q` (с использованием DDQN).
    *   Вычисляется TD-ошибка `delta = target_Q - Q(s, a)`.
    *   Вычисляется взвешенная Huber Loss (она менее чувствительна к выбросам, чем MSE).
    *   Выполняется шаг градиентного спуска для основной сети.
    *   Обновляются приоритеты в PER буфере на основе `delta`.

5.  **Выбор Действия (`SimulationWorker._select_action`)**
    *   Используется **epsilon-greedy** стратегия:
        *   С вероятностью `epsilon` выбирается случайное действие (исследование).
        *   С вероятностью `1 - epsilon` выбирается действие с максимальным Q-значением, предсказанным основной сетью (эксплуатация).
    *   `epsilon` постепенно уменьшается от `EPS_START` до `EPS_END` за `EPS_DECAY_STEPS` шагов симуляции, чтобы со временем переходить от исследования к эксплуатации знаний.

## 8. Установка и Запуск

1.  **Клонировать репозиторий:**
    ```bash
    git clone <your-repository-url>
    cd ai-life-simulator
    ```
2.  **Создать и активировать виртуальное окружение (рекомендуется):**
    ```bash
    python -m venv venv
    # Windows
    .\venv\Scripts\activate
    # macOS/Linux
    source venv/bin/activate
    ```
3.  **Установить зависимости:**
    *   Для запуска:
        ```bash
        pip install .
        ```
    *   Для запуска и разработки (включая тесты):
        ```bash
        pip install .[dev]
        ```
    *   **Примечание:** Убедитесь, что у вас установлены драйверы NVIDIA и CUDA Toolkit, совместимые с версией PyTorch (`torch==2.7.0+cu128`), если вы планируете использовать GPU.

4.  **Запустить GUI приложение:**
    ```bash
    python main.py
    ```
    (Запускает корневой `main.py`)

5.  **Запустить через скрипт (если настроен PATH):**
    ```bash
    sim_start
    ```
    (Запускает `src/ai_life_simulator/_front/cli/main.py`).

## 9. Конфигурация

*   **GUI:** Основные параметры интерфейса задаются в словаре `config` в корневом `main.py`.
*   **Симуляция и RL:** Параметры задаются в классе `Config` в `ai_life_simulator/_backend/agents.py`.
*   **Улучшение:** В будущем можно вынести параметры в отдельный конфигурационный файл (например, YAML или JSON) для удобства настройки без изменения кода.

## 10. Тестирование

*   Тесты находятся в директории `tests/`.
