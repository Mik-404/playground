# Подготовка окружения для разработки
## 1. Docker образ

Специально для вас подготовлен docker-образ, который содержит все необходимые инструменты для разработки: кросс-компилятор, утилиты, gdb, etc. Если чего-то не хватает – пишите в чат, добавим.

Чтобы получить образ, нужно для начала залогиниться в нашем docker registry:

```bash
docker login docker.gitlab.carzil.ru
```
Имя пользователя и пароль – такие же как в GitLab.

Затем добавляем в ~/.bashrc (или в ~/.zshrc, если вы пользуетесь zsh):
```bash
export PATH=$PATH:<путь-до-персонального-репозитория-с-hellos>/bin/
```

## 2. Установка QEMU
### MacOS
```
brew install qemu
```

### Ubuntu
```
apt-get install qemu-system-x86_64
```
или
```
apt install qemu-system-x86
```
## 3. Установка зависимостей Python
Шагов выше достаточно, чтобы уметь собирать и запускать HellOS. Однако скрипт тестирования требует наличия некоторых Python бибилиотек (стоит поставить python3.12). Для их установки нужно запустить:
```
./setup-deps.sh
```

После этого можно активировать virtual environment:
```
source .venv/bin/activate
```

И запускать скрипт полного тестирования:
```
python grade.py
```

# Как синхронизировать свой репозиторий с основным репозиторием HellOS?
Единожды для репозитория добавляем апстрим:
```
git remote add upstream git@gitlab.carzil.ru:os-advanced/hellos.git
```

Когда нужно обновить свой репозиторий до последней версии основного, выполняем:

```bash
git checkout main && git pull --rebase upstream main
```


# Сборка и запуск лабораторной работы
Контейнер `caos-dev` запускается один раз и работает в фоне. В нём запускается сборка, LSP, в него можно зайти через `caos-shell` и поотлаживаться изнутри. `caos-dev` запускается после ввода любой команды `caos-*`. Убедиться, что контейнер запущен:
```bash
docker container list | grep caos-dev
```
По окончании работы контейнер можно удалить с помощью `caos-stop`.

Следует иметь единственную локальную копию HellOS. Корректная работа контейнера, очевидно, не гарантируется при параллельном запуске `caos-*` из двух разных директорий.

Для сборки мы будем использовать `make`, который зашит в docker-образ. Чтобы каждый раз не писать монструозную команду для запуска команды в контейнере, написан маленький скрипт `caos-make`. Запускаете его в папке с домашкой и, вуаля, ядро собирается. Система сборки HellOS несовершенна и не всегда может инкрементально скомпилировать ядро. Поэтому лучше всего перед каждой сборкой очищать всё:
```
caos-make clean && caos-make -j<количество ядер на вашей машинке>
```

QEMU должен запускаться на машинке (не в докере!). Чтобы каждый раз не вспоминать команду, в `Makefile` есть цель `qemu`. Запустить ядро в QEMU:
```
make qemu
```
Рекомендуется посмотреть на эту команду в Makefile, чтобы в случае какого-то серьёзного дебага, можно было самому туда добавлять опции.

Если нужно запустить полное тестирование вашей версии HellOS, достаточно запустить скрипт `grade.py` (не забудьте перед этим перейти в venv, как описано выше!). Это не всегда удобно, поэтому лабы можно собирать отдельно, указав определённые флаги конфигурации в [`config_values.yaml`](./config_values.yaml). Список флагов, которые собирают различные тесты для лабораторных работ, можно найти в [`tests/config.yaml`](./tests/config.yaml).


## Запуск с отладчиком
Если надо запустить с отладчиком:
```
make qemu-gdb
```
Затем в отдельном окошке терминала запустить:
```
caos-gdb
```
Дальше у вас есть привычный GDB :). На всякий случай: [документация](https://sourceware.org/gdb/onlinedocs/gdb/index.html) и [сheat sheet](https://darkdust.net/files/GDB%20Cheat%20Sheet.pdf).

Полезные команды и хаки:
* `b _start` — брейкпоинт на функцию.
* `b kernel.c:23` — брейкпоинт на строчку в файле.
* `b 53` — брейкпоинт на строчку в текущем файле.
* `la src`/`la asm`/`la regs` — различные варианты окошек (исходный код, инструкции, регистры).
* `foc cmd`/`foc asm`/`foc src` — переключает фокус на разные окошки.
* `si` — выполнить одну инструкцию и остановиться. Полезна, когда отлаживаете ассемблерный код.
* `s` — выполнить одну строчку из файла.
* `p` — может выводить различные выражения (но не всегда все).
* `p *(struct multiboot_mmap_entry*)0x10000` — посмотреть на структуру `multiboot_mmap_entry`, которая лежит по адресу `0x10000`.
* `x` — встроенный hexdump. Прочитать про него [тут](https://sourceware.org/gdb/onlinedocs/gdb/Memory.html).

## Про LSP
Проект собирается кросс-компилятором в контейнере docker. Поэтому нужны дополнительные усилия, чтобы заработали диагностика, go-to-definition и автодополнение. Для желающих иметь эти фичи подготовлен скрипты [caos-clangd](./bin/caos-clangd) и [caos-bear](./bin/caos-bear).

[caos-clangd](./bin/caos-clangd) запускает clangd в контейнере caos-dev, создавая маппинги путей до исходников HellOS и кэша clangd.

[caos-bear](./bin/caos-bear) генерирует compile_commands.json - файл, через который clangd понимает флаги компиляции.

<details>
    <summary>VS Code</summary>

Для VSCode и его расширений:
1) Установите расширение clangd (`llvm-vs-code-extensions.vscode-clangd`).
2) Если в workspace запущен IntelliSense - отключите его. У IntelliSense не будет доступа к кросс-компилятору, его заголовкам и boost'у. Он будет подставлять заголовки системного clang/gcc, вы увидите ложные ошибки и некорректные подсказки.
3) Убедиться, что контейнер caos-dev запущен: `caos-make --version`. Контейнер запускается автоматически при любой команде caos-*.
4) Сгенерировать `compile_commands.json`: `caos-bear`.
5) Настроить расширение clangd на использование команды `caos-clangd`. Для этого в корне проекта создайте файл `.vscode/settings.json`:
```json
{
    "clangd.path": "${workspaceFolder}/bin/caos-clangd",
    "clangd.arguments": [],
    "C_Cpp.intelliSenseEngine": "disabled"
}
```
6) Перезапустить clangd: `Cmd+Shift+P` -> clangd: Restart language server
</details>

<details>
    <summary>Vim и NeoVim</summary>

Для пользователей [nvim-lspconfig](https://github.com/neovim/nvim-lspconfig):
```lua
local hellos_path = "<your-path-to-hellos>"
if starts_with(vim.fn.getcwd(), hellos_path) or vim.fn.getcwd() == hellos_path then
    clangd_cmd = { hellos_path .. "/bin/caos-clangd" }
end
lspconfig.clangd.setup({
    -- ...
    cmd = clangd_cmd,
    -- ...
})
```

Для [coc.nvim](https://github.com/neoclide/coc.nvim) пропишите в .vim/coc-settings.json (в корне HellOS) подобное содержимое:
```json
{
    "languageserver": {
        "clangd": {
            "command": "<path-to-caos-clangd>",
            "filetypes": ["c", "cpp", "h"],
            "rootPatterns": ["compile_commands.json", ".git"]
        }
    }
}
```
</details>

### info mem
Показывает текущие маппинги виртуальной памяти.
