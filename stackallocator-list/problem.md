**Предупреждение:** формально следующие задачи не будут зависеть от этой, то есть для получения баллов по любой из остальных задач не будет требоваться, чтобы эта задача была сдана. Однако код, написанный в этой задаче, пригодится для переиспользования в следующей задаче.

В этой задаче вам предлагается научиться пользоваться аллокаторами, а также разобраться с устройством контейнера `list` и понять, что иногда нестандартный аллокатор может давать выигрыш в производительности по сравнению со стандартным.

#### Часть 1. StackAllocator

Напишите класс `StackAllocator<typename T, size_t N>`, который бы позволял создавать стандартные контейнеры на стеке, без единого обращения к динамической памяти. Для этого вам пригодится написать класс `StackStorage<size_t N>`, в котором будет храниться большой массив на стеке. Объект класса `StackAllocator` должен быть легковесным, легко копироваться и присваиваться. Объект `StackStorage`, напротив, не должен поддерживать копирование.

Класс `StackAllocator` должен удовлетворять требованиям к аллокатору, описанным на [cppreference](https://en.cppreference.com/w/cpp/named_req/Allocator). Он должен быть STL-совместимым. В частности, должны быть определены:

*   Конструктор копирования, деструктор, оператор присваивания;
*   Методы `allocate`, `deallocate`;
*   Внутренний тип `value_type`;
*   Метод `select_on_container_copy_construction`, если логика работы вашего аллокатора этого потребует.

**Пример использования:**

```cpp
int main() {
    StackStorage<100000> storage;
    StackAllocator<int, 100000> alloc(storage);
    std::vector<int, StackAllocator<int, 100000>> v(alloc);
    // ... useful stuff ...
}
```

Во время исполнения может существовать несколько `StackStorage` с одним и тем же $N$. Разумеется, аллокаторы, построенные на разных `StackStorage`, должны считаться неравными.

`StackAllocator` должен заботиться о выравнивании объектов. В частности, нельзя класть переменные типа `int` по адресам, не кратным 4; переменные типа `double` — по адресам, не кратным 8, и так далее.

Использование любого контейнера со `StackAllocator` вместо `std::allocator` должно приводить к тому, что обращений к динамической памяти в программе не происходит вообще.

#### Часть 2. List

Напишите класс `List` — двусвязный список с правильным использованием аллокатора. Ваш лист должен удовлетворять требованиям [AllocatorAwareContainer](https://en.cppreference.com/w/cpp/named_req/AllocatorAwareContainer).
Должно быть два шаблонных параметра: $T$ — тип элементов в листе, `Allocator` — тип используемого аллокатора (по умолчанию — `std::allocator<T>`).

**Должны быть поддержаны:**

*   **Конструкторы:** без параметров; от одного числа; от числа и `const T&`; от одного аллокатора; от числа и аллокатора; от числа, `const T&` и аллокатора. Если создается пустой лист, то не должно быть никаких выделений динамической памяти.
*   **Метод** `get_allocator()`, возвращающий объект аллокатора.
*   **Конструктор копирования**, деструктор, копирующий оператор присваивания.
*   **Методы** `size()`, работающий за $O(1)$, и `empty()`.
*   **Методы** `push_back`, `push_front`, `pop_back`, `pop_front`.
*   **Итераторы:** двунаправленные итераторы (`BidirectionalIterator`), константные и reverse-итераторы.
*   **Методы** `begin`, `end`, `cbegin`, `cend`, `rbegin`, `rend`, `crbegin`, `crend`.
*   **Методы** `insert(iterator, const T&)` и `erase(iterator)` для одиночных элементов.

**Особенности реализации:**

1.  **Empty Base Optimization:** Если аллокатор является структурой без полей, он не должен увеличивать `sizeof` объекта листа (должен занимать 0 байт как поле).
2.  **Исключения:** Все методы листа должны быть строго безопасны относительно исключений (**strong exception safety**). Если при операции выбрасывается исключение (из методов типа $T$), лист должен вернуться в состояние до начала операции, не допуская утечек памяти и UB. Конструкторы и операторы присваивания аллокаторов исключений не кидают.

**Проверка производительности:**
Выполните тесты со случайными добавлениями и удалениями. И ваш собственный `List`, и `std::list` должны работать со `StackAllocator` быстрее, чем со стандартным аллокатором. В контесте это проверяется замерами времени: если ваш аллокатор проиграет `std::allocator`, тест не будет засчитан.
